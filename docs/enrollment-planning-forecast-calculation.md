# Enrollment Planning Forecast Calculation Trace

This document traces the current Enrollment Planning Forecast from raw enrollment inputs to the final Sustainable Growth Projection.

It is documentation only. It does not change calculations, filters, exports, state, or Room Availability behavior.

## Primary Code Path

The forecast is rendered from `js/enrollment-analytics.js`.

Main execution path:

1. `runDemand()`
2. `loadDemandRows()`
3. `demandRowsForPopulationSelections()`
4. `applyFilters(allRows, 'dem')`
5. `isComparableDemandTerm(row, target)`
6. `normalizeDemandAnalysisTerms(finalizedHistorical, target)`
7. analysis window selection from `demWindow`
8. `demandGrowthContext(rows, target)`
9. `demandForecastRowsForLevels(rows, context, growthModifier)`
10. `demandForecastRow(level, groupName, rows, context, modifierGrowth)`
11. `demandHistoricalBenchmarkProjection(termRows, currentTotals, options)`
12. shared `COSTrendProjection.buildProjection()` in `js/enrollment/trend-projection.js`

The final displayed/exported demand-growth field is `Sustainable Growth Rate`.

The diagnostic field `Projection Growth vs Historical Average` is:

```js
adjustedForecastGrowth = safeDiv(expectedEnrollmentNextTerm, avgCensusEnrollment) - 1
```

where `expectedEnrollmentNextTerm` is supplied by `demandHistoricalBenchmarkProjection()` and `avgCensusEnrollment` is the average historical census enrollment across the included comparable term buckets for the row. This diagnostic is not the growth rate used to create the projection.

## Raw Inputs

`loadDemandRows()` loads:

- manually uploaded Enrollment Planning Forecast CSV rows from `#demandCsv`
- selected archived Enrollment Planning Forecast rows from `#demArchiveTerms`
- optional Work Experience rows through `rowsWithWorkExperience(uploaded, 'dem')`

Rows are normalized by the existing enrollment normalizer before forecast logic. Relevant normalized fields include:

- `term`
- `sourceTerm`
- `division`
- `subject`
- `course`
- `section`
- `crn`
- `actual`
- `census`
- `cap`
- `ftes`
- `waitlist`
- `status`
- `modality`
- Work Experience flags such as `isWorkExperience`, `hasFtesData`, and `ftesUnavailable`

Enrollment basis in `demandTermStats()`:

```js
census = sum(row.census == null ? row.actual : row.census)
final = sum(row.actual)
```

So census enrollment is preferred when present. If a row lacks census enrollment, actual/current enrollment is used for that row's census-basis historical total.

## Population Selection

`demandRowsForPopulationSelections()` filters rows into the selected planning populations:

- Physical Campuses
- Dual Enrollment
- Work Experience

Population classification is handled by `demandPlanningPopulationType(row)`:

- Work Experience if `row.isWorkExperience` or modality is `WORK EXPERIENCE`
- Dual Enrollment if the row is identified as Dual Enrollment
- otherwise Physical Campuses

This population filter happens before report filters and before comparable historical terms are selected.

## Report Filters

After population selection, `runDemand()` applies the normal demand filters:

```js
filtered = applyFilters(allRows, 'dem')
```

Only `filtered` rows proceed into forecast target selection and term diagnostics. Terms that loaded but are removed by filters appear in diagnostics rather than being silently counted.

## Forecast Target and Comparable Series

`demandForecastTarget()` reads:

- `#demForecastScope`
- `#demForecastSeason`
- `#demForecastYear`

Single-term target:

- target season is `FALL`, `SPRING`, `SUMMER`, or `WINTER`
- target term is generated by `targetTermFromFiscalYear(season, year)`
- target sort value is based on `termSortValue(targetTerm)`

Academic-year target:

- target label is `FY/AY {year}`
- target sort value is `{year} * 10 + 5`

Comparable historical filtering is done by `isComparableDemandTerm(row, target)`.

Single-term scope:

```js
parts.season === target.season
&& termSortValue(row.term) < target.sortValue
```

Examples:

- Fall scope uses Fall-to-Fall history only.
- Spring scope uses Spring-to-Spring history only.
- Summer scope uses Summer-to-Summer history only.

Academic-year scope:

```js
academicYearTrailingYear(row.term) < target.year
&& season in [SUMMER, FALL, SPRING]
```

Then `normalizeDemandAnalysisTerms()` rewrites each included row's `term` to an annual FY/AY label for academic-year forecasts. This causes Summer + Fall + Spring rows to aggregate into annual buckets before growth is calculated.

## Analysis Window

After comparable terms are identified:

```js
selectedTerms = filteredTerms
  .sort(termSortValue)
  .slice(last demWindow terms)
```

The default UI value controls the maximum number of most recent comparable terms or FY/AY buckets. Earlier comparable terms are excluded by the analysis window.

Only `rows = analysisRows.filter(row => selectedTerms.includes(row.term))` are used for forecast calculations.

## Forecast Levels

`demandForecastRowsForLevels()` builds four levels:

- College
- Division
- Discipline
- Course

Each level calls `demandForecastRow()` with only the rows belonging to that level/group.

The final College row is the executive summary source for the overall forecast metrics. Division, Discipline, and Course rows use the same row-level projection function on their scoped rows.

## Term Aggregation

Inside `demandForecastRow()`, rows are grouped by normalized `term`:

```js
byTerm = group(rows, row => row.term || 'UNKNOWN')
termRows = demandTermStats(term, termSections)
```

`demandTermStats()` produces one term bucket per included comparable term/FY/AY:

- `sections`: row count in that term bucket
- `census`: sum of `row.census`, falling back to `row.actual` when `census` is null
- `final`: sum of `row.actual`
- `ftes`: sum of `row.ftes`
- `capacity`: sum of `row.cap`
- `waitlist`: sum of `row.waitlist`
- `fillRate`: `census / capacity`
- `finalFillRate`: `final / capacity`
- `attritionCount`: `census - final`
- `attritionRate`: `(census - final) / census`

The historical average displayed for the row is:

```js
avgCensusEnrollment = round(average(termRows.map(row => row.census)))
avgFtes = average(termRows.map(row => row.ftes))
avgCapacity = average(termRows.map(row => row.capacity))
```

## Current / Active Snapshot Totals

`demandActiveRowsForGroup()` scopes active target rows to the same College/Division/Discipline/Course group.

`demandCurrentTotals()` then calculates:

- `enrollment`: sum of census when positive, otherwise actual
- `censusEnrollment`: sum of nonnegative census
- `actualEnrollment`: sum of actual
- `seatsOffered`: sum of cap
- `scheduledClassOfferings`: distinct sections when possible, otherwise row count
- `ftes`: sum of FTES

These current totals are not the historical growth series. They are used for schedule-readiness diagnostics and confidence context. They do not reduce the enrollment-demand projection.

## Shared Trend Projection Engine

`demandHistoricalBenchmarkProjection()` calls:

```js
buildTrendProjection(termRows, currentTotals, {
  growthModifier,
  selectedGrowthSeries,
  forecastScopeLabel
})
```

`buildTrendProjection()` calls `COSTrendProjection.buildProjection()` in `js/enrollment/trend-projection.js`.

The engine normalizes each term bucket into metrics:

- enrollment
- census enrollment
- actual enrollment
- seats offered
- scheduled class offerings
- instructional meetings
- waitlist
- FTES
- student presence
- fill rate
- enrollment per offering
- seats per offering

## Growth Rows

For each projected metric, `projectMetric(termTotals, metric)` computes year-over-year rows from the selected comparable series only:

```js
rate = previous ? (current - previous) / abs(previous) : 0
```

For enrollment, the values are the selected term bucket enrollment totals in chronological order.

This means:

- Fall-to-Fall growth is calculated only from Fall term totals.
- Spring-to-Spring growth is calculated only from Spring term totals.
- Summer-to-Summer growth is calculated only from Summer term totals.
- Academic-year growth is calculated from annual FY/AY totals after Summer + Fall + Spring are aggregated into FY/AY buckets.

Fall and Spring growth rates are not summed or averaged together to create an annual growth rate.

## Recency Weighting

The trend engine creates term weights with:

```js
recencyWeights(length, floor = 0.35)
```

For multiple terms, the oldest term weight starts at `0.35` and linearly increases to `1.00` for the newest term. If there is only one term, its weight is `1`.

Growth-rate weights use `weights.slice(1)` because a growth rate exists between two terms, not on the first term itself.

The final recency-weighted growth rate is:

```js
weightedGrowth =
  sum(growthRate[index] * growthWeight[index])
  / sum(growthWeight[index])
```

If the denominator is zero, the helper falls back to an unweighted average of the available values.

The trend projection before schedule adjustment is:

```js
trendProjection = max(0, mostRecentValue * (1 + weightedGrowth))
```

The audit stores this as:

- `finalGrowthRateUsed`
- `projectionBeforeScheduleAdjustment`
- growth rows with rate, weight, and weighted contribution

## Schedule Adjustment

The shared trend engine calculates:

```js
offeringFactor = currentOfferings / historicalBaselineOfferings
seatFactor = currentSeats / historicalBaselineSeats
combinedFactor = (offeringFactor * 0.55) + (seatFactor * 0.45)
```

If historical or current offerings/seats are unavailable, each factor falls back to `1`.

The first schedule-adjusted projection is:

```js
scheduleAdjustedEnrollment = enrollmentTrendProjection * combinedFactor
```

Then the manual growth modifier is applied:

```js
finalExpectedEnrollment = scheduleAdjustedEnrollment * (1 + growthModifier)
```

This is the uncapped model output returned from the shared trend engine.

## Sustainable Growth Projection Wrapper

`demandHistoricalBenchmarkProjection()` wraps the shared trend engine and applies COS planning assumptions:

- start from the most recent comparable completed term or FY/AY as the previous comparable benchmark
- calculate a sustainable growth rate from comparable historical growth
- cap unusually high recovery-spike growth rates at `10%` when they exceed the `12%` recovery-spike threshold
- floor routine negative historical growth rates at `0%`
- treat current schedule size as supply/readiness context rather than as a demand reducer
- build a bounded planning range around the most-likely estimate

Intermediate values:

- `historicalAverageEnrollment`: average census across included comparable buckets
- `baselineEnrollment`: most recent comparable completed census enrollment, falling back to historical average
- `sustainableGrowthRate`: recency-weighted average of dampened growth rows
- `scheduleAdjustment`: legacy/shared trend schedule factor retained as schedule-readiness diagnostics
- `scheduleSupplyReadinessFactor`: loaded current schedule size compared with historical schedule size
- `scheduleAdjustmentUsedAsDemandFactor`: always `false` for this planning projection
- `uncappedGrowth`: `uncappedMostLikely / baselineEnrollment - 1`

Growth row dampening is:

```js
nonRegressionRate = max(0, rawGrowthRate)
adjustedRate =
  nonRegressionRate > 0.12
    ? min(nonRegressionRate, 0.10)
    : nonRegressionRate
```

Then:

```js
mostLikely = max(
  baselineEnrollment,
  baselineEnrollment * (1 + sustainableGrowthRate + growthModifier)
)
```

This `mostLikely` value becomes `trendModel.finalExpectedProjection.enrollment`.

## FTES Projection

FTES growth is calculated in the shared trend engine as its own metric, using the same selected comparable series and recency weighting method.

However, the final displayed FTES projection in `demandHistoricalBenchmarkProjection()` is ratio-based:

```js
historicalFtesPerEnrollment = baselineFtes / baselineEnrollment
mostLikelyFtes =
  historicalFtesPerEnrollment > 0
    ? mostLikely * historicalFtesPerEnrollment
    : baselineFtes
```

So the final FTES estimate is tied to the final most-likely enrollment projection and the historical FTES-per-enrollment ratio.

The FTES range applies the same enrollment range spread through that ratio.

## Final Sustainable Growth Output

Back in `demandForecastRow()`, the final displayed/exported row values are assigned.

The forecasted enrollment is:

```js
expectedEnrollmentNextTerm =
  round(trendModel.finalExpectedProjection.enrollment)
```

The final sustainable growth rate is:

```js
sustainableGrowthRate = trendModel.sustainableGrowthRate
```

The projection-growth diagnostic versus the historical average is:

```js
adjustedForecastGrowth =
  safeDiv(expectedEnrollmentNextTerm, avgCensusEnrollment) - 1
```

This is important:

- `finalEnrollmentGrowthRateUsed` is the shared trend engine's raw recency-weighted growth rate before schedule-readiness context.
- `recencyWeightedGrowth` is the shared trend engine enrollment growth rate.
- `sustainableGrowthRate` is the dampened planning growth rate used for the demand projection.
- `uncappedAdjustedForecastGrowth` is the wrapper's uncapped growth versus the previous comparable benchmark.
- `adjustedForecastGrowth` / `Projection Growth vs Historical Average` is the final rounded forecast enrollment compared with the average historical census enrollment for that row.

Because it is derived from final expected enrollment divided by average historical census enrollment, `Projection Growth vs Historical Average` reflects:

1. selected comparable growth series
2. recency-weighted growth
3. sustainable-growth dampening
4. manual growth modifier
5. non-regression floor
6. rounding of the final expected enrollment

## Legacy Blended Growth Context

`demandGrowthContext()` still calculates:

- `collegeGrowth`
- division growth map
- discipline growth map
- course growth through `demandTrend()`

`blendedForecastGrowth()` still exists and blends:

- College row: college growth + modifier
- Division row: 70% division + 30% college + modifier
- Discipline row: 60% discipline + 25% division + 15% college + modifier
- Course row: 50% course + 20% discipline + 15% division + 15% college + modifier

This blended value is currently a fallback path inside `demandForecastRow()` if `trendModel.finalExpectedProjection` is unavailable.

In the normal current path, `demandHistoricalBenchmarkProjection()` returns a trend model, so the final expected enrollment and final `adjustedForecastGrowth` come from the historical benchmark projection path described above. The context growth values remain in the row for diagnostics and table/export columns.

## Worked Formula Chain

For each forecast row:

```text
Raw rows
-> population selection
-> report filters
-> comparable terms only
-> most recent N terms/FY buckets from analysis window
-> group by forecast level
-> group by term
-> termRows census/FTES/capacity/sections totals
-> shared trend projection:
     term totals sorted chronologically
     growth rates between adjacent comparable terms
     recency weights from 0.35 to 1.00
     recency-weighted growth rate
     most recent total x (1 + weighted growth)
     schedule readiness factor
     manual growth modifier
-> sustainable growth wrapper:
     previous comparable benchmark = most recent completed comparable term/FY bucket
     cap recovery-spike growth rates above 12% to 10%
     floor routine negative growth rates at 0%
     sustainable growth rate from dampened growth rows
     current schedule size remains readiness context only
     final most-likely enrollment
     FTES via historical FTES/enrollment ratio
-> row output:
     expectedEnrollmentNextTerm
     sustainableGrowthRate
     adjustedForecastGrowth = expectedEnrollmentNextTerm / avgCensusEnrollment - 1 (diagnostic)
```

## Verification Notes

The existing test suite includes checks that:

- same-season or FY/AY-only growth series are documented and audited
- Fall and Spring growth rates are not combined into one annual rate
- expected FTES range mirrors enrollment projection range behavior
- trend projection replaces simple historical averages
- schedule readiness does not reduce projected enrollment demand

Relevant tests are in `tests/enrollment-analytics-smoke.test.js`, especially the forecast growth audit and demand projection tests.
